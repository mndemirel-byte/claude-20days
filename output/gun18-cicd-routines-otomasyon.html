BÖLÜM 1TEORİK TEMEL

1.1 Headless Mode ve Agent SDK CLI

Claude Code'un asıl gücü interaktif terminalde değil, non-interactive modda ortaya çıkar. claude -p (veya --print) bir prompt alır, çalışır, çıktıyı döner ve çıkar — bir session açmaz. Bu sayede CI pipeline'ında, pre-commit hook'ta veya herhangi bir bash script'te Claude Code'u tıpkı grep veya jq gibi bir Unix aracı olarak kullanabilirsin.

# Basit non-interactive kullanım
claude -p "Explain what this project does"

# Structured output — script/jq ile parse
claude -p "List all API endpoints" --output-format json

# Streaming — gerçek zamanlı işleme
claude -p "Analyze this log file" --output-format stream-json --verbose

stdin pipe desteği sayesinde build loglarını, diff çıktılarını veya herhangi bir metin akışını doğrudan Claude'a besleyebilirsin. Pipe girişi 10 MB ile sınırlı — daha büyük inputları dosya olarak yaz ve dosya yolunu prompt'ta referans ver.

# Build log analizi — stdin pipe
cat build-error.txt | claude -p 'concisely explain the root cause of this build error' > output.txt

# Git diff'i review'e gönder
git diff main | claude -p 'review these changes for security issues'

--output-format json kullandığında response payload'ı total_cost_usd ve model bazında maliyet dağılımı içerir — scripted çağrılarda harcamayı usage dashboard'a gitmeden izleyebilirsin.

1.2 Availability vs Permission: Flag'leri Doğru Anlamak

Claude Code CLI'da tool erişimini kontrol eden beş flag var ve bunlar iki farklı katmanda çalışır. Bu ayrımı bilmeden CI job tasarlamak güvenlik yanılsaması yaratır:

FlagKatmanNe yaparNe yapmaz
--toolsBuilt-in tool availabilityClaude'un kullanabileceği built-in tool setini sınırlar (ör. "Read,Glob,Grep" → sadece bunlar mevcut)MCP tool'larını etkilemez
--allowedToolsPermission allow ruleBelirtilen tool/pattern için izin sormadan çalıştırırTool availability kısıtlamaz — listelenmemiş tool hâlâ mevcut, sadece prompt ister
--disallowedToolsDeny rule / availabilityBare tool name → tool'u context'ten kaldırır; scoped pattern → o pattern'i bloklarMCP tool'lar için ayrıca "mcp__*" gerekir
--permission-mode dontAskRuntime approval policyPre-approved olmayan tool çağrılarını reddederAllowlist'teki tool'ların güvenliğini garanti etmez
--bareStartup/context minimizationCustomization discovery atlar: hooks, skills, plugins, MCP, auto memory, CLAUDE.md yüklenmezTool erişimini kısıtlamaz (Bash/Read/Edit hâlâ açık)
🔑 ÖnemliCI'da read-only Claude job'u tasarlamanın doğru formülü: --bare (öngörülebilir startup) + --tools "Read,Glob,Grep" (built-in availability kısıtla) + --disallowedTools "mcp__*" (MCP bypass engelle) + --permission-mode dontAsk (unlisted her şeyi reddet). Tek bir flag 'güvenlik modu' değildir; katmanları birlikte kullanman gerekir.

--bare modu, CI ve script çağrılarında context yüzeyini küçültmek, startup davranışını öngörülebilir yapmak ve project/user customization etkisini azaltmak için kullanılır. Hangi makinede çalıştırırsan çalıştır, aynı sonucu alırsın — bir takım arkadaşının ~/.claude'ındaki hook veya projedeki .mcp.json'daki MCP server çalışmaz. Ama --bare tek başına güvenlik sandbox'ı sağlamaz: Bash, file read ve file edit tool'ları açık kalır.

⚠️ Dikkat--bare artık scripted ve SDK çağrıları için önerilen moddur ve gelecekte -p kullanıldığında varsayılan olacak. CI pipeline'ında --bare yoksa, proje dizinindeki hooks, MCP server'lar ve CLAUDE.md yüklenebilir — bu hem tutarsız sonuçlara hem de güvenlik riskine yol açabilir.

Son olarak, --continue ve --resume flag'leri ile non-interactive çağrıları zincirlemenin mümkün olduğunu bil. Session ID'yi JSON çıktısından yakalayıp bir sonraki çağrıda kullanabilirsin — çok aşamalı CI pipeline'ları için faydalı.

💡 İpucuAgent SDK'nın Python ve TypeScript paketleri de mevcut; structured output, tool approval callback ve native message nesneleri sunarlar. CLI'daki --tools'un SDK karşılığı da tools opsiyonudur. Detay: code.claude.com/docs/en/agent-sdk

1.3 GitHub Entegrasyonları — Net Ayrım

Claude Code'un GitHub ile çalışmasının dört farklı yolu var. Bunlar farklı altyapılarda, farklı tetikleyicilerle çalışır ve farklı soruları cevaplar. İkisini karıştırmak — özellikle Code Review ile Actions'ı — CI pipeline tasarımında en yaygın hatadır:

AraçNerede çalışırTetiklemeTipik kullanımKullanılabilirlik
GitHub Code ReviewAnthropic managed infraPR open / push / @claude reviewDeep multi-agent PR review, inline findingTeam/Enterprise (ZDR'da yok)
GitHub ActionsGitHub runner (self/cloud)issue/PR mention, workflow eventCustom automation, implementation, CI stepTüm planlar (API key gerekir)
/autofix-prClaude Code on the web sessionCurrent branch PR + CI/review eventsCI failure ve reviewer comment fix loopclaude.ai subscription + gh CLI
Local /code-reviewDeveloper machineManuel commandLocal diff review, günlük sürüşTüm yüzeyler

GitHub Code Review, Anthropic'in yönettiği bir servistir: PR açıldığında (veya push/manual @claude review ile) bir filo uzmanlaşmış agent, değişiklikleri full codebase bağlamında inceler ve inline comment olarak bulgularını bırakır. Severity seviyeleri: Important (ciddi bug veya güvenlik açığı), Nit (stil veya küçük iyileştirme), Pre-existing (değişiklikten önce zaten olan sorun). Her comment'te 👍/👎 butonları gelir — bu feedback Anthropic'in reviewer'ı iyileştirmesi için kullanılır.

CLAUDE.md'nin yanı sıra proje kökünde bir REVIEW.md dosyası oluşturabilirsin — Code Review bunu en yüksek öncelikle okur. Nit hacmini sınırlama, belirli path'leri atlama, repo-spesifik kontrol kuralları eklemek için kullan.

GitHub Actions ise kendi runner'ında Claude Code Action çalıştırır. /install-github-app ile kurulur; @claude mention ile PR/issue'da tetiklenir veya workflow event'le çalışır. Custom automation, feature implementation, CI step olarak kullanılabilir. 3P provider (Bedrock/Vertex) desteği var.

/autofix-pr ise mevcut checked-out branch'teki open PR'ı gh pr view ile bulur, bir Claude Code on the web session'ı başlatır ve PR'daki CI failure ve reviewer comment'lere otomatik tepki verir. Net fix'leri uygular, belirsiz durumlarda sana sorar.

🔑 ÖnemliCode Review = 'PR'ıma bağımsız gözlerle baksın'; Actions = 'Claude ile custom otomasyonu tetikleyeyim'; /autofix-pr = 'CI failure'ları otomatik fix etsin'. Üçü birbirinin yerine geçmez, birbirini tamamlar.

1.4 Zamanlama Üç Katmanı

Claude Code'da tekrarlayan veya zamanlanmış iş çalıştırmanın üç farklı katmanı var. Doğru katmanı seçmek verimliliği ve güvenilirliği doğrudan etkiler:

KatmanNerede çalışırTetiklemeYaşam süresiEn iyi kullanım
/loopMevcut CLI session içiCron expression veya doğal dilSession-scoped; recurring 7 gün expiry5-60 dk geçici polling (deploy, CI check)
Desktop scheduled taskLokal makine (Desktop app)Cron / one-off / doğal dilApp açıkken; catch-up on wakeLocal dosya/tool erişimi gereken periyodik iş
Cloud RoutineAnthropic managed infraCron / one-off / API / GitHub eventLaptop kapalıyken bile çalışırRepo/connector gerektiren otonom, tekrarlayan iş

Seçim heuristiği şu sorularla başlar: Laptop kapalıyken çalışmalı mı? → Cloud Routine. Local dosyalara erişmeli mi? → Desktop scheduled task. Sadece bu session'da geçici bir polling mi? → /loop. CI event kaynaklı deterministic iş mi? → GitHub Actions. PR review kalitesi ve inline finding mi? → Code Review managed.

Desktop scheduled task'lar bilgisayar uyandığında son 7 gündeki kaçırılmış run'lardan yalnızca en güncel olanı çalıştırır — eski missed run'lar tek tek yakalanmaz. 6 gün uyuduysa, tek bir catch-up run yapılır. Task'ın kendi permission mode'u vardır; Ask/manual mode'da kurulursa tool approval bekleyip stall olabilir — ilk 'Run now' ile izinleri gözlemle, 'always allow' kuralları tanımla.

Cloud Routines ise tamamen farklı bir modelde çalışır: Anthropic altyapısında, otonom bir Claude Code cloud session olarak yürütülür. Permission-mode picker ve approval prompt yoktur — bu yüzden scope'u (repo, branch-push, environment, connector) dikkatli sınırla. Varsayılan olarak yalnızca claude/ prefix'li branch'lara push edebilir.

⚠️ DikkatRoutines research preview'dadır; plan/organization policy ve daily run allowance kısıtlamaları var. Routine actions, bağlı GitHub/Slack identity üzerinden 'kullanıcı adına' görünür — commit'ler senin GitHub kullanıcınla, Slack mesajları senin hesabınla gider. Routine prompt'unu self-contained yaz ve repo/connector scope'unu minimum tut.
💡 İpucuCLI'dan claude /schedule ile doğal dille routine oluşturabilirsin: '/schedule her pazartesi 9'da açık PR'ları incele'. Web (claude.ai/code/routines) ve Desktop app'ten de oluşturulabilir — üç yüzey aynı cloud hesabına yazar.

1.5 Chrome ve Slack Entegrasyonları

Chrome entegrasyonu, Claude Code'u canlı web uygulamasının console log'larına, DOM'una ve network request'lerine bağlar. Debugging döngüsü: console error → kaynak dosya tespit → fix → browser'da doğrulama. Claude in Chrome browser extension'ı gerektirir, Chrome ve Edge'de çalışır, WSL'de desteklenmez. /chrome komutuyla bağlantıyı yönetirsin.

Slack entegrasyonuyla @Claude mention, bir coding task'ı otomatik tespit edip Claude Code on the web session'ı başlatır. Routing mode: Code + Chat — yanlış yönlenirse 'Retry as Code' veya 'Retry as Chat' butonlarıyla düzelt. Sadece channel'larda (public/private) çalışır, DM'lerde çalışmaz.

⚠️ DikkatClaude Code in Slack, Team/Enterprise tarafında Claude Tag ile değiştiriliyor. Aynı Slack app, yeniden kurulum gereksiz; ama kesin UI ve kurulum adımları plan/tenant'a göre değişebilir. Bugün amacımız: Slack'ten coding intent ile web session delegation modelini anlamak, spesifik kurulum yapmak değil.
⚠️ DikkatSlack thread context'i bir prompt injection yüzeyi oluşturabilir — Claude, thread'deki diğer mesajlardan gelen yönlendirmeleri takip edebilir. @Claude'u yalnızca güvenilir kanallarda kullan.

1.6 CI Güvenlik Modeli

CI ortamında Claude Code çalıştırırken güvenlik tek bir flag'le değil, katmanların birleşimiyle sağlanır:

  • --bare: Predictability — hooks, MCP, plugins, auto memory yüklenmez; hangi runner'da çalışırsa çalışsın aynı sonuç
  • --tools "Read,Glob,Grep": Built-in availability — Claude sadece belirtilen tool'ları kullanabilir (Edit, Write, Bash mevcut değil)
  • --disallowedTools "mcp__*": MCP bypass engeli — --tools MCP tool'ları etkilemez, bu flag onları kapatır
  • --permission-mode dontAsk: Runtime safety net — pre-approved olmayan her çağrı reddedilir
  • Runner/sandbox isolation: GitHub Actions runner veya Docker container ile OS seviyesinde izolasyon
  • Secret yönetimi: ANTHROPIC_API_KEY GitHub Secrets'ta saklanır; --bare modda OAuth/keychain okunmaz, kimlik doğrulama ANTHROPIC_API_KEY env var veya apiKeyHelper ile yapılır
🔑 Önemli--bare öngörülebilirlik sağlar ama güvenlik garantisi vermez. --tools built-in availability kısıtlar ama MCP'yi kapamaz. --permission-mode dontAsk run-time net oluşturur ama allowlist'teki tool'ların kendisi zararlı olabilir. Güvenlik = katmanların birleşimi + runner izolasyonu + secret yönetimi.

2.1 İlk Non-Interactive Çalıştırma

En basit kullanımla başla: projenin kök dizininde claude -p ile bir codebase özeti al.

  1. SaaS Dashboard proje dizinine geç.
  2. Şu komutu çalıştır: claude -p "Summarize the project structure and list the main API endpoints"
  3. Çıktıyı incele — Claude projeyi okudu, analiz etti ve sonucu döndü; interaktif session açmadan çıktı.
  4. Şimdi aynı komutu --output-format json ile tekrar çalıştır ve JSON çıktısındaki result, total_cost_usd ve session_id alanlarını incele.
# JSON çıktılı çalıştırma
claude -p "List all API endpoints" --output-format json

# jq ile parse
claude -p "List all API endpoints" --output-format json | jq '.result'

2.2 --bare + --tools ile Read-Only Review

Şimdi CI'da kullanacağın read-only review pattern'ini oluştur. Amacımız: Claude sadece dosyaları okusun, hiçbir şey düzenlemesin, hiçbir komut çalıştırmasın.

  1. Aşağıdaki komutu çalıştır — --tools built-in tool setini Read/Glob/Grep ile sınırlar; --disallowedTools "mcp__*" MCP bypass'ı engeller:
  2. --permission-mode dontAsk ile Claude allowlist dışındaki hiçbir tool'u çalıştıramaz — en sıkı runtime kontrolü.
  3. Çıktıdaki findings'i incele. JSON formatında file, line, risk ve recommendation alanlarını gözlemle.
claude --bare -p "Review this repository for OWASP Top 10 risks. Do not edit files. Return findings with file, line, risk, evidence, and recommendation." \
  --tools "Read,Glob,Grep" \
  --disallowedTools "mcp__*" \
  --permission-mode dontAsk \
  --output-format json
💡 İpucuBu pattern'i bir bash script'e sarıp .github/scripts/claude-review.sh olarak kaydet — GitHub Actions workflow'unda bu script'i çağırabilirsin.

2.3 stdin Pipe ile Build Log Analizi

  1. Kasıtlı bir build hatası oluştur (ör. bir import'u kır).
  2. Build'ı çalıştır ve çıktıyı bir dosyaya kaydet: npm run build 2>&1 > build-error.txt (veya projenin build komutu)
  3. Build logunu Claude'a pipe et: cat build-error.txt | claude -p 'concisely explain the root cause of this build error' > analysis.txt
  4. analysis.txt'i aç ve Claude'un root cause analizini incele.
⚠️ DikkatPipe girişi 10 MB ile sınırlıdır. Daha büyük log dosyaları için içeriği bir dosyaya yaz ve prompt'ta dosya yolunu referans ver: claude -p 'Analyze the build log at ./build-error.txt'

2.4 Permission Mode Deneyimi

--permission-mode seçeneklerini karşılaştırarak run-time davranış farkını gör:

  1. Aynı review prompt'unu --permission-mode dontAsk ile çalıştır — allowlist dışındaki her tool call reddedilir.
  2. Şimdi --permission-mode acceptEdits ile dene — dosya yazma izni otomatik verilir, ama diğer shell komutları hâlâ prompt ister (non-interactive'de abort eder).
  3. Farkı gözlemle: dontAsk en sıkı (CI için ideal), acceptEdits dosya düzenleme otomasyonu için uygun.
# dontAsk: sadece allowedTools çalışır, geri kalan denied
claude --bare -p "Apply lint fixes" \
  --permission-mode dontAsk \
  --allowedTools "Read,Edit,Bash(npm run lint *)" \
  --output-format json

# acceptEdits: dosya yazma otomatik, shell hâlâ kısıtlı
claude --bare -p "Apply lint fixes" \
  --permission-mode acceptEdits

2.5 GitHub Actions Workflow Yazma

PR'da otomatik Claude Code review çalıştıran basit bir GitHub Actions workflow dosyası oluştur.

  1. Proje kökünde .github/workflows/claude-review.yml dosyası oluştur.
  2. Workflow'u pull_request event'ine bağla.
  3. İki job tanımla: (1) standart lint/test, (2) Claude Code review — böylece code quality ve AI review ayrı raporlanır.
  4. Claude job'unda ANTHROPIC_API_KEY'i GitHub Secrets'tan oku.
  5. Çıktıyı gh pr comment ile PR'a yaz veya workflow artifact olarak kaydet.
# .github/workflows/claude-review.yml (basitleştirilmiş)
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  lint-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  claude-review:
    runs-on: ubuntu-latest
    needs: lint-test
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Install Claude Code
        run: |
          curl -fsSL https://claude.ai/install.sh | bash
      - name: Run review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          git diff origin/main...HEAD | \
          claude --bare -p 'Review this diff for bugs, security issues, and improvements. Format as markdown.' \
            --tools "Read,Glob,Grep" \
            --disallowedTools "mcp__*" \
            --permission-mode dontAsk \
            > review.md
      - name: Post comment
        run: gh pr comment ${{ github.event.number }} -F review.md
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
⚠️ DikkatBu basitleştirilmiş bir örnektir. Prodüksiyonda Claude Code GitHub Actions (anthropics/claude-code-action) kullanmak daha iyi entegrasyon ve güvenlik sağlar. Ayrıca managed Code Review (Team/Enterprise) @claude review ile daha derin, multi-agent analiz yapar — self-hosted pipeline'dan farklı bir üründür.

2.6 Desktop Scheduled Task Kurma

  1. Desktop app'te Routines → New routine → Local seç.
  2. Name: 'dependency-audit', Instructions: 'Check for outdated dependencies, list any with known vulnerabilities, and suggest update commands.'
  3. Schedule: her gün sabah 09:00.
  4. Permission mode: 'Accept edits' (veya daha güvenli 'Ask' — ama ilk run'da izinleri 'always allow' ile tanımla).
  5. Folder: SaaS Dashboard proje dizini.
  6. 'Run now' ile test et ve çıktıyı incele.

Alternatif olarak doğal dille de oluşturabilirsin — herhangi bir session'da şunu yaz: 'Set up a daily code review that runs every morning at 9am'. Claude otomatik olarak bir scheduled task oluşturur.

💡 İpucuTask prompt'unu ~/.claude/scheduled-tasks/<task-name>/SKILL.md dosyasından düzenleyebilirsin. YAML frontmatter'da name ve description, body'de prompt bulunur.

2.7 Chrome Debug Demo

Bu adım gösterimli bir demhodur — Chrome extension kurulumu ve site permission'lar ortama bağlıdır. WSL kullanıyorsan bu adımı atla.

  1. Terminal'de --chrome flag'iyle veya session içinde /chrome komutuyla Chrome entegrasyonunu aktif et.
  2. Claude'a şu prompt'u ver: 'Open localhost:3000, fill the login form with test credentials, submit, and check the console for errors.'
  3. Claude'un Chrome'da tab açıp, formu doldurup, console loglarını okuyup, varsa hata kaynağını tespit etmesini gözlemle.
  4. Eğer hata bulunursa, Claude'dan düzeltme yapmasını iste ve browser'da doğrulamasını izle.
🔑 ÖnemliChrome entegrasyonunun değeri 'code → browser → console → source → fix' döngüsünü tek session'da kapatmasıdır. Context switch'i ortadan kaldırır.

BÖLÜM 3PRATİK — DERİNLEŞ

3.1 Cloud Routine Kurma

Desktop task'tan farklı olarak cloud Routine, Anthropic'in yönettiği bir VM'de çalışır — laptop kapalı olsa bile. Şimdi CLI'dan bir haftalık bağımlılık denetimi routine'i kuracaksın.

  1. Terminal'de şu komutu çalıştır: /schedule every Monday at 09:00, inspect package.json for outdated or vulnerable dependencies, summarize findings, and post to the configured Slack connector
  2. Claude seni GitHub bağlantısı ve cloud environment kurulumu için yönlendirecek — /web-setup çalıştırman gerekebilir.
  3. Routine oluşturulduktan sonra claude.ai/code/routines sayfasında veya Desktop app'in Routines sekmesinde görüntüle.
  4. 'Run now' ile test et ve session transcript'ini oku.
⚠️ DikkatCloud Routine'ler autonomous çalışır — permission-mode picker ve approval prompt yoktur. Scope'u kontrol et: hangi repo'lar bağlı, branch-push açık mı, hangi connector'lar mevcut. Varsayılan olarak sadece claude/ prefix'li branch'lara push edebilir.

3.2 GitHub Event Tetikli Routine

Schedule dışında, Routine'leri GitHub event'leriyle de tetikleyebilirsin — örneğin PR açıldığında otomatik bir review oturumu başlatmak.

  1. claude.ai/code/routines → yeni routine oluştur veya mevcut routine'i düzenle.
  2. Trigger olarak 'GitHub event' seç → pull_request.opened.
  3. Filter ekle: base branch = 'main', head branch contains 'feature'.
  4. Prompt: 'Review this PR for correctness and security issues. Leave inline comments on findings.'
  5. Kaydet ve bir test PR aç — routine'in tetiklenip tetiklenmediğini session listesinden kontrol et.
💡 İpucuFilter'da matches regex operatörü tüm field değerini test eder; substring match için contains kullan. Örn. .*hotfix.* regex ile, hotfix contains ile — ikisi aynı sonucu verir ama regex syntax hatası riski taşır.

3.3 Slack Delegation Demo

Bu adım gösterimli bir demhodur — Slack app kurulumu workspace admin gerektirir ve plan/tenant'a bağlıdır.

  1. Slack channel'ında @Claude ile bir bug report yaz: '@Claude the /api/reports endpoint returns 500 when date range is empty. Fix it in the backend repo and create a PR.'
  2. Claude'un coding intent'i tespit edip bir Claude Code on the web session'ı başlattığını gözlemle.
  3. Slack thread'inde progress update'ler görünecek; tamamlandığında 'View Session' ve 'Create PR' butonları gelecek.
  4. Session'ı aç ve Claude'un yaptığı değişiklikleri incele.

3.4 End-to-End CI Pipeline Tasarımı

Şimdiye kadar öğrendiğin parçaları bir araya getirerek production-ready bir CI pipeline akışı tasarla:

KÜTÜPHANE🧰 Prompt & Şablon Kütüphanesi

CI Read-Only Review
claude --bare -p "Review this repository for OWASP Top 10 risks. Do not edit files. Return findings with file, line, risk, evidence, and recommendation." \
  --tools "Read,Glob,Grep" \
  --disallowedTools "mcp__*" \
  --permission-mode dontAsk \
  --output-format json
--tools built-in availability kısıtlar, --disallowedTools MCP bypass'ı engeller, --permission-mode dontAsk unlisted her şeyi reddeder. Üç katman birlikte read-only garanti eder.
Otomatik Commit
claude -p "Review staged changes and create a conventional commit message, then commit." \
  --allowedTools "Bash(git diff *),Bash(git log *),Bash(git status *),Bash(git commit *)"
Local/dev ortamı için uygundur. CI'da otomatik commit/push çok daha sıkı branch protection ve review gate ister. Bash(git diff *) 'taki boşluk + * prefix match sağlar — boşluksuz yazmak git diff-index'e de izin verir.
Build Log Analiz
cat build-error.txt | claude -p 'concisely explain the root cause of this build error' > output.txt
stdin pipe 10 MB ile sınırlı. Daha büyük inputlar için dosya yolunu prompt'ta referans ver. --output-format json ile total_cost_usd alabilirsin.
Cloud Routine — Haftalık Dependency Audit
/schedule every Monday at 09:00, inspect the selected repository for outdated or vulnerable dependencies, summarize findings in a markdown report, and post to the configured Slack connector. Do not push changes unless the routine is explicitly configured with branch-push permission.
Routine autonomous çalışır — permission prompt yok. Prompt'u self-contained yaz: 'başarılı sonuç neye benzer' ve 'sonuçla ne yap' açık olmalı, çünkü çalışırken sana soru soramaz.
Chrome Debug
Open localhost:3000, reproduce the form submission error, inspect console logs, identify the source file, propose a minimal fix, apply it, and verify in the browser.
Chrome entegrasyonu --chrome flag'i veya /chrome komutu ile aktif edilir. Chrome/Edge destekli, WSL'de desteklenmez. Plan mode'da read-only browser tool'lar (read_page, get_page_text) prompt gerekmez; state-changing tool'lar onay ister.

CHALLENGE🏆 Günün Challenge'ı

CHALLENGE: Safe Claude CI Review Pipeline

SaaS Dashboard için güvenli bir CI pipeline oluştur: PR açıldığında lint/test çalıştıran bir GitHub Actions workflow yaz, ayrı bir job'da Claude Code ile read-only diff analizi yap, API key'i güvenli şekilde yönet ve sonucu PR'a raporla.

Gereksinimler:

  • .github/workflows/claude-review.yml oluştur; pull_request event'ine bağla.
  • İki ayrı job: (1) lint-test — standart npm test, (2) claude-review — AI-powered diff analizi.
  • Claude job'unda --bare, --tools "Read,Glob,Grep", --disallowedTools "mcp__*" ve --permission-mode dontAsk kullan.
  • ANTHROPIC_API_KEY'i GitHub Secrets'tan oku (${{ secrets.ANTHROPIC_API_KEY }}).
  • Çıktıyı gh pr comment, actions/github-script veya workflow artifact olarak PR'a raporla.
  • 'Pipeline failure' (test kırıldı) ile 'review finding' (Claude bulgu raporladı ama CI yeşil) ayrımını README veya workflow comment'inde açıkla.

Başarı Kriterleri:

✅ Çözüm Rehberi

Bu challenge'ın çekirdeği güvenli bir non-interactive CI job tasarlamaktır. İlk adım workflow YAML iskeletini oluşturmak, ikinci adım Claude job'unu doğru flag'lerle kilitlemek, üçüncü adım çıktıyı PR'a raporlamaktır.

Önerilen prompt zinciri:

1) Workflow YAML İskeleti
SaaS Dashboard için bir GitHub Actions workflow oluştur: PR açıldığında (1) npm test çalıştır, (2) ayrı bir job'da claude --bare -p ile git diff'i review et. ANTHROPIC_API_KEY GitHub Secrets'tan okunmalı. Çıktıyı gh pr comment ile PR'a yaz.
2) Read-Only Flag Kombinasyonu
claude --bare -p ile bir review komutu yaz. Şartlar: sadece Read, Glob, Grep tool'ları mevcut olsun (--tools), MCP tool'ları devre dışı (--disallowedTools "mcp__*"), izin istemeden çalışsın ama allowlist dışı reddedilsin (--permission-mode dontAsk). JSON çıktı versin.
3) Pipeline Failure vs Finding Açıklaması
README'ye bir bölüm ekle: 'CI pipeline'da test failure (CI kırmızı, merge bloklanır) ile Claude review finding (CI yeşil kalır, yalnızca bilgilendirme) arasındaki fark nedir?' Teknik olmayan bir PM'in anlayacağı şekilde yaz.
4) (Bonus) Routine Kurulumu
/schedule every Monday at 09:00, inspect this repository for outdated dependencies and known CVEs, summarize in markdown, post to Slack.

Kontrol noktaları:

  • Workflow YAML'ı yazarken fetch-depth: 0 ile tam git history alınmalı — aksi halde git diff origin/main...HEAD boş dönebilir.
  • GitHub Secrets'a ANTHROPIC_API_KEY eklemeyi unutma — workflow bunu bulamazsa Claude adımı sessizce fail olur.
  • Claude job'unu needs: lint-test ile bağlarsan, test kırıldığında review çalışmaz — bu genellikle istenen davranıştır.
  • gh pr comment komutu GITHUB_TOKEN env var gerektirir; Actions'ın otomatik sağladığı secrets.GITHUB_TOKEN yeterlidir.

Sık yapılan hatalar:

  • ⚠️ --allowedTools'un tool availability kısıtladığını sanmak — bu flag sadece permission katmanında çalışır; availability için --tools kullan.
  • ⚠️ --bare yokken CI'da çalıştırmak — proje dizinindeki hooks, MCP server'lar ve CLAUDE.md öngörülemeyen davranışlara yol açar.
  • ⚠️ Desktop scheduled task'ı Ask mode'da kurup tool approval stall yaşamak — ilk 'Run now' ile izinleri 'always allow' olarak tanımla.
  • ⚠️ Bash(git diff*) yazıp boşluk koymamak — boşluksuz yazımda git diff-index gibi istenmeyen komutlara da izin verilir.
  • ⚠️ Cloud Routine scope'unu kontrol etmemek — routine bağlı GitHub identity'nle commit atar ve Slack mesajı senin adına gider.
  • ⚠️ Code Review severity'lerini yanlış bilmek — Important/Nit/Pre-existing, 🔴🟠🟡 gibi emoji değil.

ÖZET📌 Takeaway'ler

  1. claude -p ile non-interactive çalıştırma, Claude Code'u CI/CD pipeline'ına, pre-commit hook'a ve bash script'e entegre etmenin temel mekanizmasıdır; stdin pipe ve --output-format json ile Unix araç zincirinin doğal bir parçası olur.
  2. Tool erişim kontrolü iki katmanlıdır: --tools built-in availability kısıtlar (Edit, Write, Bash mevcut değil), --allowedTools sadece permission katmanında çalışır (listelenmemiş tool hâlâ mevcut, sadece izin ister). MCP tool'ları --tools tarafından etkilenmez — --disallowedTools "mcp__*" ayrıca gerekir.
  3. --bare güvenlik modu değil, öngörülebilir startup modudur: hooks, skills, plugins, MCP, auto memory ve CLAUDE.md yüklenmez, ama Bash/Read/Edit tool'ları açık kalır. CI'da güvenlik = --bare + --tools + --disallowedTools + --permission-mode dontAsk + runner isolation.
  4. GitHub Code Review (managed, multi-agent, inline finding) ile GitHub Actions (self-hosted, custom workflow) farklı ürünlerdir. Code Review severity'leri: Important, Nit, Pre-existing. İkisi birbirinin yerine geçmez, birbirini tamamlar.
  5. Zamanlama üç katmandır: /loop session-scoped geçici polling, Desktop scheduled task local dosya/tool erişimli periyodik iş, Cloud Routine laptop kapalıyken bile çalışan otonom cloud session. Seçim kriteri: 'nerede çalışmalı, ne kadar yaşamalı, neye erişmeli.'
  6. Cloud Routine'ler autonomous çalışır — permission prompt ve approval mekanizması yok. Actions bağlı kullanıcı kimliğiyle görünür. Scope'u (repo, branch-push, connector) minimum tut ve prompt'u self-contained yaz.
  7. Chrome entegrasyonu 'code → browser → console → source → fix' döngüsünü tek session'da kapatır; Slack entegrasyonu coding intent'i tespit edip web session başlatır — ikisi de core CI değil, tamamlayıcı otomasyon yüzeyleridir.
  8. CI pipeline'da AI review, insan review'ını tamamlar, yerine geçmez. Code Review inline finding'leri PR'ı approve veya block etmez — son karar her zaman mühendistedir.
