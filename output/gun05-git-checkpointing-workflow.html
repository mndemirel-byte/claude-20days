Git Entegrasyonu, Checkpointing ve Oturum Yönetimi

“Claude Code, Git'in en iyi arkadaşı”

Bu üçlü, Claude Code'u 'tek seferlik kod yazdırma aracı' olmaktan çıkarıp profesyonel geliştirme workflow'una dönüştürür.

1.1 Büyük Resim: Üç Güvenlik Katmanı

Claude Code ile çalışırken üç farklı güvenlik mekanizması seni korur. Her birinin kapsamı ve kalıcılığı farklıdır; birbirinin yerine geçmez, birbirini tamamlar.

KatmanNe korur?KalıcılıkGeri alma

🔑 ÖnemliGit proje tarihçesini korur; checkpoint çalışma anındaki AI denemelerini geri alır; session yönetimi ise konuşma bağlamını sürdürür. Bu üçlüyü birbirine karıştırma — her biri farklı bir sorunu çözer.

1.2 Claude Code ve Git: Nasıl Çalışır?

Claude Code, Git işlemlerini çoğunlukla terminal komutları üzerinden yürütür. Doğal dilde verdiğin talimatı uygun git komutlarına çevirir, izin moduna bağlı olarak çalıştırır ve sonucu sana raporlar. Bu yüzden yaptığı işlemleri git status, git diff, git log gibi komutlarla doğrulamak iyi bir alışkanlıktır.

  • git status / diff / log: Okuma işlemleri — genelde izin gerekmez.
  • git add / commit: Yazma işlemleri — izin moduna göre onay gerekebilir.
  • git push / pull / merge: Ağ ve birleştirme işlemleri — yüksek etkili, genelde onay gerektirir.
  • git checkout / branch: Branch yönetimi — düşük riskli, auto mode'da çoğunlukla otomatik.
💡 İpucuClaude Code bir GUI git client değil. Arka planda terminal komutları çalıştırır. Bu avantajdır: Claude'un yaptığı her işlem standart git komutları olduğu için sonuçları her zaman git log, git diff veya VS Code Git panelinde doğrulayabilirsin.

1.3 Commit Stratejileri ve Mesaj Yazdırma

İyi bir commit stratejisi, proje tarihçesini okunabilir ve geri alınabilir kılar. Claude Code ile iki temel strateji kullanabilirsin:

StratejiNe zaman?AvantajRisk
Atomic commitHer mantıksal değişiklik ayrı commitGeri alma kolay, PR review temizDaha fazla commit, daha sık onay
Batch commitBirkaç ilişkili değişiklik tek commitHızlı, az gürültüGeri alma zor, review karışık

Conventional Commit formatı takımlar arası standart bir dil sağlar:

type(scope): kısa açıklama

[opsiyonel gövde]

[opsiyonel footer: BREAKING CHANGE, Closes #123]
TipAnlamÖrnek

fixHata düzeltmefix(api): handle null response in /users
refactorDavranış değiştirmeden yapı iyileştirmerefactor(db): extract query builder
docsDokümantasyondocs(readme): add setup instructions
testTest ekleme/düzeltmetest(auth): add edge case for expired token

💡 İpucuCommit mesajını İngilizce mi Türkçe mi yazmalı? Takım standardına bağlı. Açık kaynak veya uluslararası takımlarda İngilizce tercih edilir. Claude Code'a CLAUDE.md'de dil tercihi belirtirsen otomatik uyar.

1.4 Branch Workflow

Feature branch pattern, paralel geliştirmenin temel taşıdır. Akış şöyle işler:

  1. main branch'ten yeni bir feature branch oluştur (git checkout -b feature/add-priority)
  2. Feature branch'te çalış, commit'ler at
  3. İş bitince main'e dön ve feature branch'i merge et (git merge feature/add-priority)
  4. Conflict varsa çöz, merge commit'i oluştur
  5. Feature branch'i sil (git branch -d feature/add-priority)

Merge conflict iki branch'te aynı dosyanın aynı bölümü farklı değiştirildiğinde oluşur. Git hangi değişikliğin geçerli olduğuna karar veremez ve conflict marker'ları ile seni bilgilendirir:

<<<<<<< HEAD
function formatTodo(todo) {
  return `[${todo.priority}] ${todo.title}`;
}
=======
function formatTodo(todo) {
  return `${todo.title} (due: ${todo.dueDate})`;
}
>>>>>>> feature/add-duedate
🔑 ÖnemliClaude Code conflict marker'larını okur, her iki tarafın amacını analiz eder, uygun bir birleştirme önerir ve uygular. Ama sonucu mutlaka kendin doğrula — Claude her iki değişikliğin iş mantığını her zaman tam kavrayamayabilir.

1.5 Checkpointing: Oturum İçi Güvenlik Ağı

Checkpointing, Claude Code'un dosya düzenleme araçları (file edit tools) ile yaptığı her değişikliği otomatik olarak kaydeder. Her kullanıcı prompt'u yeni bir checkpoint oluşturur. Checkpoint'ler oturumlar arası da erişilebilir kalır — resumed bir session'da bile önceki checkpoint'lere dönebilirsin.

/rewind menüsü altı aksiyon sunar:

AksiyonDosyalarKonuşmaNe zaman kullan?
Restore code and conversation⬅ Geri alınır⬅ Geri alınırHem kodu hem konuşmayı o noktaya döndürmek istiyorsan
Restore conversationDeğişmez⬅ Geri alınırKod doğru ama konuşma yanlış yöne gittiyse
Restore code⬅ Geri alınırDeğişmezKonuşma doğru ama kod bozulduysa
Summarize from hereDeğişmezSeçilen noktadan sonrası özetlenirGereksiz konuşmayı sıkıştırıp context kazanmak
Summarize up to hereDeğişmezSeçilen noktadan öncesi özetlenirErken tartışmayı sıkıştırıp son çalışmayı korumak
Never mindDeğişmezDeğişmezVazgeçtim, menüden çık
⚠️ DikkatCheckpoint'in izlemediği üç kritik durum: 1. Bash komutları ile dosya değişiklikleri izlenmez. Claude rm, mv, cp çalıştırırsa /rewind bu değişiklikleri geri alamaz. 2. Dışarıdan yapılan değişiklikler izlenmez. Editörde veya başka terminalde yaptığın değişiklikler checkpoint kapsamı dışındadır. 3. Checkpoint, versiyon kontrol yerine geçmez. Checkpoint'i 'lokal undo', Git'i 'kalıcı tarihçe' olarak düşün.
💡 İpucu/clear sonrası bile /rewind menüsü önceki session'a dönme seçeneği sunar (menüde "/resume <session-id> (previous session)" girişi). Bu özellik Claude Code v2.1.191 veya sonrası gerektirir; eski sürümlerde /resume ile önceki session'ı seçebilirsin.

1.6 Oturum (Session) Yönetimi

Claude Code her konuşmayı otomatik olarak diske kaydeder (~/.claude/projects/<proje>/<session-id>.jsonl). Bu sayede bir göreve dönebilir, alternatif bir yaklaşım deneyebilir veya görevler arası geçiş yapabilirsin.

Komut / FlagİşlevTipik kullanım
claude --continueSon oturumu devam ettirTerminal kapandı, kaldığın yerden devam
claude --resumeOturum seçici açBirden fazla görev arası geçiş
claude --resume <isim>İsimle doğrudan devamParalel çalışmada hızlı erişim
claude --from-pr <no>PR'a bağlı oturumu bulPR review'dan oturuma dönme
/resumeAktif session içinden geçişGörev değiştirme
/branch [isim]Oturumu dallandır (fork)Alternatif yaklaşım denemesi
/exportKonuşmayı dışa aktarArşivleme, paylaşma
/rename <isim>Oturumu isimlendirSession picker'da kolay bulma
claude -n <isim>Başlangıçta isim verParalel görevlerde ayrışma
🔑 ÖnemliSession'ları isimlendirmek (claude -n veya /rename) paralel çalışmada hayati önem taşır. 'claude --resume auth-refactor' yazmak, session picker'da eski konuşmaları taramaktan çok daha hızlıdır.

Session picker kısayolları: Ctrl+W = tüm worktree'leri göster, Ctrl+A = makinedeki tüm projeleri göster, Ctrl+B = mevcut branch'e filtrele, Ctrl+R = session'ı yeniden adlandır, Space = önizleme.

2.1 İlk Commit: Diff Gör, Onayla, Commit Et

Claude Code ile commit atmanın güvenli yolu: önce değişiklikleri görmek, sonra onaylamak. Bu 'güvenlik kapısı' alışkanlığını bugünden başlat.

  1. Claude Code'u Todo App dizininde başlat: claude
  2. Bir değişiklik yap: "README.md'ye proje açıklaması ve kurulum talimatları ekle"
  3. Commit isteği ver — ama önce diff görmek iste: "Değişiklikleri analiz et. Önce git status ve git diff özetini göster. Ben onaylamadan commit atma. Commit mesajını conventional commit formatında öner."
  4. Claude diff'i gösterecek ve commit mesajı önerecek — onayladığında commit atar
  5. Sonucu doğrula: git log --oneline -3
# Claude'un oluşturacağı tipik çıktı:
$ git log --oneline -3
a1b2c3d docs(readme): add project description and setup instructions
f4e5d6c feat(init): initialize todo-app project structure
1234567 Initial commit
🔑 Önemli"Önce diff göster, onaylamadan commit atma" alışkanlığı profesyonel workflow'un temelidir. CLAUDE.md'ye bu kuralı ekleyerek Claude Code'un her zaman bu akışı izlemesini sağlayabilirsin.

2.2 Feature Branch Oluşturma ve Geliştirme

  1. Claude'a branch oluşturmasını söyle: "feature/add-priority adında yeni bir branch oluştur ve geçiş yap"
  2. Claude'un çalıştıracağı komutu ve çıktıyı gözlemle
  3. Branch'te bir özellik implement et: "Todo'lara priority (low/medium/high) alanı ekle. Mevcut todo oluşturma ve listeleme fonksiyonlarını buna göre güncelle."
  4. Değişiklikleri commit et: "Bu değişiklikleri atomic commit'ler halinde commit et. Her mantıksal değişiklik ayrı commit olsun."
# Branch oluşturma çıktısı:
$ git checkout -b feature/add-priority
Switched to a new branch 'feature/add-priority'

# Sonuç:
$ git log --oneline feature/add-priority ^main
b7c8d9e feat(todo): add priority field to todo model
e0f1a2b feat(todo): display priority in list view

2.3 Deterministik Merge Conflict Oluşturma ve Çözme

Conflict'i rastlantıya bırakmayacağız. Adım adım, kontrollü bir conflict oluşturup Claude Code ile çözeceğiz.

  1. main branch'e dön: "main branch'e geçiş yap" → git checkout main
  2. main'de aynı dosyayı değiştir: "src/todo.js (veya .py) dosyasındaki formatTodo fonksiyonunu düzenle: çıktıya oluşturulma tarihini ekle" (Claude main'de bu değişikliği yapar ve commit eder)
  3. feature branch'i merge et: "feature/add-priority branch'ini main ile merge et"
  4. Conflict gözlemle: formatTodo fonksiyonu her iki branch'te farklı değiştirildiği için Git conflict oluşturacak
  5. Claude'dan çözüm iste: "Bu merge conflict'i çöz. Her iki tarafın amacını analiz et: main'deki tarih alanını ve feature'daki priority alanını birleştir."
  6. Sonucu doğrula: Claude çözüm uyguladıktan sonra dosyayı aç ve her iki özelliğin de (tarih + priority) korunduğunu kontrol et
💡 İpucuConflict çözdükten sonra testi çalıştırmayı unutma. Claude'a "merge conflict'i çözdükten sonra testleri çalıştır" dersen hem çözüm hem doğrulamayı tek adımda halleder.

2.4 /rewind ile Geri Sarma Pratiği

  1. Claude'a kasıtlı bir hatalı değişiklik yaptır: "formatTodo fonksiyonunun return ifadesini sil ve yerine boş string döndür"
  2. Değişikliğin yapıldığını doğrula — dosyayı editörde aç
  3. Geri sar: Esc tuşuna iki kez bas (prompt boşken) veya /rewind yaz
  4. Rewind menüsünde hatalı prompt'u bul, seç
  5. "Restore code and conversation" seçeneğini seç
  6. Dosyayı tekrar aç — formatTodo fonksiyonunun eski haline döndüğünü gör
⚠️ DikkatEğer Claude hatalı değişikliği Bash komutuyla yaptıysa (örneğin sed ile), /rewind bunu geri alamaz. Bu durumda git checkout -- <dosya> veya git stash kullan.

BÖLÜM 3PRATİK — DERİNLEŞ / UYGULAMA

3.1 Oturum Yönetimi Pratiği

Aşağıdaki alıştırmalar session yönetimini günlük workflow'una yerleştirir. Her adımı sırayla dene.

  1. Oturumu isimlendir: /rename todo-priority yazarak mevcut session'a isim ver
  2. Çıkış yap: Ctrl+C ile Claude Code'dan çık
  3. Devam et: claude --continue ile son oturuma dön — context'in korunduğunu gör
  4. Tekrar çık ve bu sefer claude --resume ile oturum seçiciyi aç
  5. Kısayolları dene: Ctrl+W ile worktree'leri göster, Space ile önizle
  6. İsimle devam et: claude --resume todo-priority ile doğrudan dön
💡 İpucuParalel çalışma senaryosu: bir terminalde 'claude -n backend-api' ile backend, diğerinde 'claude -n frontend-ui' ile frontend geliştir. İki session birbirinden bağımsızdır ve ikisi de isimle resumable.

3.2 /branch ile Oturum Dallandırma

Farklı bir yaklaşım denemek istiyorsun ama mevcut çalışmayı kaybetmek istemiyorsun. /branch tam bunu yapar: konuşmayı kopyalar, yeni bir session açar, orijinali korur.

  1. Todo App session'ında /branch try-streaming yaz
  2. Yeni branch'te alternatif implementasyon dene: "Todo listesini streaming response ile gösterecek şekilde yeniden yaz"
  3. Sonuç beğenmediysen: /resume todo-priority ile orijinal session'a dön
  4. Sonuç beğendiysen: yeni session'da devam et — orijinal hâlâ session picker'da
🔑 Önemli/branch konuşmayı dallandırır, dosya sistemini değil. Dallanmış session'da yapılan dosya değişiklikleri gerçektir ve aynı dizinde çalışan diğer session'lar tarafından görülür. Dosya izolasyonu istiyorsan worktree kullan (Gün 16).

3.3 PR Oluşturma (GitHub CLI ile)

Claude Code, GitHub CLI (gh) kuruluysa doğrudan PR oluşturabilir. Bu bölüm gh kurulu olmayanlar için opsiyoneldir.

  1. Feature branch'e geçiş yap ve push et: "feature/add-priority branch'ine geçip remote'a push et"
  2. PR oluştur: "Bu branch için bir PR oluştur. PR açıklamasında ne değiştiğini, neden değiştiğini ve test talimatlarını yaz."
  3. PR sonrası oturuma dönme pratiği: terminal kapat, sonra claude --from-pr 1 ile PR'a bağlı session'ı bul
⚠️ DikkatPR pratiği için gerekli ön koşullar: (1) gh kurulu ve gh auth login yapılmış, (2) remote repository tanımlı (git remote -v), (3) branch push edilmiş, (4) diff ve test review yapılmış. Bunlar hazır değilse bu bölümü atla — diğer pratikler gh gerektirmez.

3.4 .gitignore Yönetimi

Claude Code ile .gitignore yönetimi doğal dil kadar basit. Ama hangi dosyaların ignore edilmesi gerektiğini bilmek senin sorumluluğun.

  1. Claude'a: ".gitignore dosyasına node_modules, .env, __pycache__, .claude/ worktree dizinleri ve IDE ayarlarını ekle"
  2. Sonucu doğrula: cat .gitignore
  3. CLAUDE.md'ye .gitignore kuralı ekle: "Yeni dosya oluştururken .gitignore'u kontrol et ve gerekiyorsa güncelle"
💡 İpucu.claude/worktrees/ dizinini .gitignore'a ekle — worktree içerikleri main checkout'ta untracked dosya olarak görünmesini istemezsin.

KÜTÜPHANE🧰 Prompt & Şablon Kütüphanesi

Güvenli Commit (Diff Gate)
Değişiklikleri analiz et. Önce git status ve git diff özetini göster. Mantıksal olarak ayrılabilecek değişiklikler varsa commit planı öner. Ben onaylamadan commit atma. Commit mesajlarını conventional commit formatında yaz.
Bu prompt 'güvenlik kapısı' mantığıyla çalışır: önce görüntüle, sonra onayla. CLAUDE.md'ye kalıcı kural olarak eklenmesi önerilir.
Branch Workflow Başlatıcı
feature/<isim> adında yeni bir branch oluştur, bu branch'e geçiş yap, ve mevcut proje durumunu kısaca bildir (hangi branch'teyiz, kaç commit farkımız var).
Conflict Çözücü
Bu branch'i main ile merge et. Conflict oluşursa her iki tarafı analiz et, her iki değişikliğin amacını koru ve uygun bir birleştirme uygula. Çözüm sonrası testleri çalıştır.
Claude conflict marker'larını okuyup çözüm önerir, ama sonucu mutlaka doğrula.
PR Hazırlayıcı
Son commit'lerdeki değişiklikleri analiz et. PR açıklaması yaz: ne değişti, neden değişti, nasıl test edilir. Sonra gh cli ile PR oluştur.
gh CLI kurulu ve auth yapılmış olmalı. Yoksa sadece PR açıklamasını yazdır.
Git Dedektifi
Son 10 commit'i analiz et. Hangi dosyalar en çok değişmiş, commit pattern'leri nasıl, conventional commit formatına uyum oranı nedir? Kısa bir rapor çıkar.
Mevcut projenin git tarihçesini anlamak ve iyileştirme noktalarını bulmak için.

CHALLENGE🏆 Günün Challenge'ı

Çok-Branch'li Geliştirme ve Rewind Simülasyonu

Todo App projesinde çok-branch'li bir geliştirme simülasyonu gerçekleştir. Üç farklı feature'ı branch'lerde geliştir, merge et, en az bir conflict çöz ve /rewind ile geri sarma pratiği yap.

Gereksinimler:

  • Todo App projesinde 3 feature branch oluştur: feature/add-priority, feature/add-duedate, feature/add-categories
  • Her branch'te ilgili özelliği Claude Code ile implement et
  • Her branch'te en az 1 commit (conventional commit formatında) at
  • Branch'leri sırayla main'e merge et — en az bir merge conflict oluştur ve Claude Code ile çöz
  • Bir branch'teki değişikliği bilinçli olarak boz, /rewind ile geri sar
  • Tüm merge'ler sonrası 'git log --oneline --graph --all' ile branch/merge tarihçesini incele
  • Session'ı isimlendir ve --resume ile geri dönmeyi dene

Başarı Kriterleri:

✅ Çözüm Rehberi

Bu challenge'ı adım adım çözmek için aşağıdaki akışı izle. Önemli olan sıralama: branch'leri sırayla merge etmen gerekir ki conflict oluşsun.

Önerilen prompt zinciri:

1. Priority branch
feature/add-priority branch'i oluştur. Todo model'ine priority alanı (low/medium/high) ekle, listeleme ve oluşturma fonksiyonlarını güncelle.
2. Conflict tohumu (main)
main'e dön. src/todo dosyasındaki formatTodo fonksiyonuna created_at tarihi ekle ve commit et.
3. İlk merge (conflict çöz)
feature/add-priority'yi main ile merge et. Conflict varsa her iki tarafı (priority + tarih) koru.
4. Due date branch
feature/add-duedate branch'i oluştur. Todo'lara due date alanı ekle.
5. Categories branch
feature/add-categories branch'i oluştur. Todo'lara kategori (work/personal/urgent) ekle.
6. Bilinçli bozma + rewind
Bir değişikliği kasıtlı olarak boz: formatTodo fonksiyonunu tamamen sil.
7. Kalan merge'ler
Kalan branch'leri sırayla main ile merge et.

Kontrol noktaları:

  • Conflict'i garanti etmek için main'de ve feature branch'te aynı fonksiyonun aynı satırlarını değiştir. Önerilen hedef: formatTodo (veya eşdeğeri) fonksiyonu.
  • Merge sırası önemlidir — ilk merge conflict yaratır, sonrakiler genelde temiz geçer (farklı dosya/satır).
  • /rewind pratiğini merge'lerden önce yap ki geri sarılacak temiz bir checkpoint olsun.

Sık yapılan hatalar:

  • ⚠️ Bash ile dosya silme (rm, mv) checkpoint'te izlenmez — /rewind bunları geri alamaz. Bu durumda 'git checkout -- <dosya>' kullan.
  • ⚠️ Merge conflict çözdükten sonra test çalıştırmayı unutma — Claude iki tarafı birleştirse bile iş mantığı bozulmuş olabilir.
  • ⚠️ PR oluşturmak için gh CLI kurulu ve auth yapılmış olmalı — kurulu değilse sadece PR açıklaması yazdır ve commit'leri push etmekle yetin.
  • ⚠️ Session'ları isimlendirmezsen /resume menüsünde hangi konuşmanın hangisi olduğunu ayırt edemezsin.
  • ⚠️ /branch dosya sistemini dallandırmaz, sadece konuşmayı dallandırır — iki dallanmış session aynı dosyaları düzenleyebilir.

ÖZET📌 Takeaway'ler

  1. Git kalıcı proje tarihçesi, checkpoint oturum içi geri alma, session konuşma bağlamı sürdürme — üçü birbirini tamamlar, yerine geçmez
  2. Claude Code git komutlarını terminal üzerinden yürütür; sonuçları git log, git diff ile doğrulamak her zaman iyi pratiktir
  3. "Önce diff göster, onaylamadan commit atma" güvenlik kapısı profesyonel workflow'un temelidir
  4. Conventional commit formatı (feat/fix/refactor/docs/test/chore) takım iletişimini standartlaştırır
  5. /rewind menüsü 6 aksiyon sunar: 3 restore + 2 summarize + never mind — her birinin farklı kullanım senaryosu var
  6. Checkpoint yalnızca Claude'un file edit tool değişikliklerini izler; Bash komutları ve dışarıdan yapılan değişiklikler kapsam dışıdır
  7. Session'ları isimlendirmek (claude -n veya /rename) paralel çalışmada zaman kazandırır
  8. /branch konuşmayı dallandırır ama dosya sistemini değil — dosya izolasyonu için worktree gerekir
