Subagent'lar: Uzmanlaşmış AI Takım Arkadaşları

“Tek Claude yerine, uzman bir takım kur”

Hafta 3'e hoş geldin. Şimdiye kadar tek bir Claude Code oturumuyla çalıştın — her şey aynı context window'da biriktı. Bugün bunu değiştiriyorsun: subagent'lar, kendi context penceresi, kendi system prompt'u ve kendi tool/izin setiyle çalışan uzmanlaşmış asistanlardır. Ana konuşmanı log/dosya/arama çıktısıyla kirletmeden bir işi devretmeyi, ne zaman devredeceğine karar vermeyi ve güvenli (least-privilege) bir agent tasarlamayı öğreneceksin.

BÖLÜM 1TEORİK TEMEL

1.1 Subagent Nedir?

Bir subagent, ana konuşmandan bağımsız kendi context window'unda çalışan, kendi system prompt'u ve kendi tool/izin setiyle donatılmış özel bir asistandır. Ana Claude, bir alt görevin (onlarca dosya okuma, kapsamlı arama, log analizi gibi) ana konuşmayı gereksiz detayla dolduracağını fark ettiğinde işi bir subagent'a devredebilir — subagent kendi context'inde çalışır, ana konuşmaya yalnızca özet sonucu döner.

🔑 ÖnemliSubagent kullanmanın asıl gerekçesi context izolasyonudur: 20 dosya okuyup sonuç çıkaran bir araştırma, subagent olmadan yapılırsa o 20 dosyanın tamamı ana context'te kalıcı olarak birikir. Subagent ile ana konuşma sadece özeti görür.

1.2 Built-in Subagent'lar

Claude Code, kendi agent dosyanı yazmadan önce kullanabileceğin hazır subagent'larla gelir. Bunlardan Explore ve Plan, hız için CLAUDE.md ve git status yüklemeyi atlar — bu onları hafif keşif görevleri için uygun, ama proje-özel kurallara ihtiyaç duyan görevler için eksik kılar.

AgentAmaçNot
ExploreKod tabanını hızlıca tarama, dosya/desen bulmaCLAUDE.md ve git status yüklemez — hız için
PlanDeğişiklik öncesi plan çıkarma, tasarım keşfiCLAUDE.md ve git status yüklemez — hız için
general-purposeGenel amaçlı, geniş tool erişimli görevlerModel, ana oturumun modeliyle sınırlıdır (üst sınır: Opus)

1.3 Kapsam ve Öncelik Sırası

Aynı isimli birden fazla agent tanımı varsa, Claude Code aşağıdaki sırayla en yüksek önceliklisini kullanır. Ayrıca proje kapsamındaki .claude/agents/ klasörü sadece repo kökünde değil, çalışma dizininden yukarı doğru taranır; aynı isim birden fazla iç içe .claude/agents/ klasöründe tanımlıysa, çalışma dizinine en yakın olan kazanır ("closest wins").

ÖncelikKaynakKapsam
1 (en yüksek)Managed settingsOrganizasyon tarafından zorunlu kılınan agent'lar
2--agents CLI flag'iOturum başlatılırken elle belirtilen agent'lar
3.claude/agents/ (proje)Repo/çalışma dizini kapsamlı, takımla paylaşılır
4~/.claude/agents/ (kullanıcı)Tüm projelerde kişisel olarak kullanılır
5 (en düşük)Plugin agent'larıBir plugin/marketplace ile birlikte gelir

1.4 Frontmatter Anatomisi

Bir agent dosyası, YAML frontmatter + markdown gövdeden oluşur. Gövde, subagent'ın system prompt'u olur. Frontmatter'da yalnızca name ve description zorunludur; geri kalan her şey opsiyoneldir ve verilmezse mantıklı bir varsayılana düşer.

---
name: code-reviewer
description: Kod değişikliklerini güvenlik ve kalite açısından inceler. Değişiklik yapıldıktan sonra proaktif olarak kullan.
tools: Read, Grep, Glob
model: sonnet
permissionMode: default
---
Sen kıdemli bir kod inceleme uzmanısın. İncelediğin her değişiklikte:
1. Güvenlik açıklarını (injection, secret sızıntısı) kontrol et
2. Hata yönetimi eksiklerini işaretle
3. Bulgularını önem sırasına göre listele, kod değiştirme
🔑 Önemlitools alanı verilmezse subagent, parent (ana) oturumun erişebildiği tüm tool setini devralır — bu "her şeyi yapabilir" anlamına gelmez, parent'ın zaten erişebildiği tool setiyle sınırlıdır. Yine de bir subagent'ı belirli bir işe kilitlemek istiyorsan tools alanını açıkça yaz.

1.5 tools vs permissionMode — İkisi Aynı Şey Değil

tools (ve disallowedTools), subagent'ın hangi araçlara erişebileceğini belirleyen bir yetenek listesidir (allowlist/denylist). permissionMode ise bu araçları kullanırken onay isteme davranışını belirler. Bir subagent'ın tools listesinde Bash olması, permissionMode ayarına göre her komut için onay isteyebilir ya da hiç sormadan çalışabilir — ikisi farklı katmanlardır.

AlanNe belirlerÖrnek
tools / disallowedToolsHangi araçlara erişim var (yetenek kapsamı)tools: Read, Grep, Glob
permissionModeAraç kullanılırken onay davranışıdefault, acceptEdits, auto, dontAsk, bypassPermissions, plan
💡 İpucupermissionMode güncel değerleri: default (standart onay akışı), acceptEdits (dosya düzenlemelerini otomatik kabul et), auto, dontAsk, bypassPermissions (tüm onayları atla — dikkatli kullan) ve plan (sadece planla, uygulama). manual, v2.1.200+ sürümlerde default için bir alias olarak eklendi.

1.6 Model Çözümleme Sırası

Bir subagent hangi modelle çalışacağına şu sırayla karar verir: önce CLAUDE_CODE_SUBAGENT_MODEL ortam değişkeni kontrol edilir, sonra çağrı anında verilen bir model parametresi, sonra agent dosyasının frontmatter'ındaki model alanı, ve hiçbiri yoksa ana oturumun kullandığı model kullanılır.

🔑 ÖnemliModel çözümleme sırası: CLAUDE_CODE_SUBAGENT_MODEL → invocation model → frontmatter model → ana oturum modeli. Bu sırayı bilmek, 'neden bu agent beklediğimden farklı bir modelle çalıştı' sorusunun cevabıdır.

1.8 Subagent mı, Agent Teams mı?

Subagent'lar tek bir oturum içinde çalışır ve sonucu doğrudan ana konuşmaya (parent'a) raporlar. Agent Teams (Gün 17'de işlenecek) ise birbirinden bağımsız birden fazla Claude Code oturumunu, paylaşılan bir görev listesi ve doğrudan ajanlar-arası mesajlaşma ile koordine eder — farklı bir mimari katmandır.

İhtiyaçKullan
Ana konuşmayı kirletmeden araştırma/review yaptırmakSubagent
Tek oturum içinde uzman görev delegasyonuSubagent
Birden fazla bağımsız Claude Code oturumunu koordine etmekAgent Teams
Agent'ların birbiriyle mesajlaşması / paylaşılan görev listesiAgent Teams
⚠️ Dikkat/agents komutu artık interaktif bir sihirbaz açmıyor (v2.1.198+). Yeni bir agent oluşturmak için ya Claude'a doğal dille yazdırırsın ("bana X yapan bir subagent oluştur") ya da .claude/agents/ altına dosyayı elle eklersin.

BÖLÜM 2PRATİK — ADIM ADIM

2.1 İlk Agent: code-reviewer

İlk agent'ın salt-okunur olmalı — bu, agent tasarımında least-privilege (en az yetki) prensibinin en basit uygulamasıdır: bir kod inceleme agent'ının dosya değiştirmesine gerek yoktur, o yüzden ona bu yetkiyi hiç vermeyiz.

  1. Todo App proje dizinine geç ve Claude Code'u başlat: claude
  2. Agent oluşturma promptunu ver (Prompt Kütüphanesi'ndeki 1. prompt)
  3. Oluşan .claude/agents/code-reviewer.md dosyasını aç, frontmatter'ı incele
  4. tools alanının sadece Read, Grep, Glob içerdiğini doğrula — Write/Edit/Bash yok
⚠️ Dikkat.claude/agents/ klasörü oturum başladıktan sonra ilk kez oluşturuluyorsa (daha önce hiç yoktuysa) çalışan oturum bunu algılamaz — Claude Code'u yeniden başlatman gerekir. Klasör zaten varsa, içine eklenen/değiştirilen dosyalar birkaç saniye içinde algılanır, restart gerekmez.

2.2 Agent'ı Çağırma

Bir subagent'ı kullanmanın üç farklı kesinlik seviyesi vardır: doğal dilde önerme, @-mention ile garanti etme, ve --agent ile tüm oturumu o agent kimliğiyle başlatma. Bugün ilk ikisini kullanacaksın.

  1. Doğal dille dene: "code-reviewer agent'ını kullanarak son değişiklikleri incele"
  2. Claude'un bunu bir subagent çağrısı olarak tanıdığını gözlemle
  3. Aynı işi @-mention ile garanti ederek tekrar dene: @code-reviewer son değişiklikleri incele
  4. İki çağırma yönteminin sonucunu karşılaştır
YöntemKesinlikKapsam
Doğal dilClaude uygun görürse delegasyon yapar (garanti değil)Tek görev, öneri niteliğinde
@-mentionBelirtilen subagent'ın bu görev için çalışması garanti edilirTek görev, kesin
claude --agent <name>Tüm oturum o agent'ın system prompt/tool/model ayarıyla başlarOturum genelinde (session-wide)
🔑 Önemli--agent, tek görevlik bir çağrı değildir — ana oturumun kendisini o agent'ın kimliğiyle (system prompt, tool kısıtlaması, model) başlatır. Bunu, 'bu oturum boyunca ben artık code-reviewer'ım' demek gibi düşün.

BÖLÜM 3PRATİK — DERİNLEŞ

3.1 İkinci Agent: test-writer

code-reviewer'ın tersine, test-writer dosya yazması ve testleri çalıştırması gereken bir agent'tır — ama ona sınırsız Bash vermek yerine, sadece gerekli komutlara izin veren kısıtlı bir tool kuralı yazacaksın. Bu, tools alanının sadece açık/kapalı değil, komut-özel kısıtlama da destekleyebildiğini gösterir.

---
name: test-writer
description: Eksik test kapsamını bulur ve test yazar. Yeni bir endpoint/fonksiyon eklendiğinde proaktif olarak kullan.
tools: Read, Grep, Glob, Edit, Bash(pytest *), Bash(npm test *)
permissionMode: default
---
Sen bir test yazma uzmanısın. Görevin:
1. Test kapsamı eksik olan fonksiyon/endpoint'leri bul
2. Edge case'leri (boş girdi, olmayan id, geçersiz payload) kapsayan testler yaz
3. Yazdığın testleri çalıştırıp geçtiğini doğrula
💡 İpucuBash(pytest *) gibi komut-özel kurallar, agent'a genel Bash erişimi vermek yerine sadece belirli komutlara izin verir — sınırsız Bash verilen bir test-writer teorik olarak rm -rf da çalıştırabilir, kısıtlı kural bunu engeller.

3.2 Zincirleme: reviewer → tester

İki agent'ı art arda çağırarak zincirleme delegasyon deneyeceksin: önce code-reviewer sorunları bulsun, sonra test-writer o sorunları kapsayan testler yazsın. Her iki subagent da kendi izole context'inde çalışır; ana konuşma sadece özet sonuçları görür.

  1. "code-reviewer ile son değişiklikleri incele" de, bulguları oku
  2. Bulunan sorunlardan en az birini kapsayan bir test istekle: "test-writer ile bu sorunu kapsayan bir test yaz"
  3. Testin gerçekten çalıştığını doğrula

3.3 Nested Subagent'lar ve Derinlik Limiti

Bir subagent, kendi içinden başka bir subagent çağırabilir — buna nested (iç içe) subagent denir. Ama bunun gerçekleşmesi için o subagent'ın tools listesinde Agent tool'unun bulunması gerekir. Bir subagent'ın başka subagent doğurmasını engellemek istiyorsan Agent'ı tools listesine hiç koyma ya da disallowedTools içine ekle.

⚠️ DikkatNested subagent zincirinin bir derinlik limiti vardır: beşinci seviyedeki bir subagent artık Agent tool'unu almaz ve daha fazla subagent doğuramaz — bu, sonsuz/kontrolsüz agent zincirlerini önleyen bir güvenlik sınırıdır.

3.4 Foreground / Background Davranışı

v2.1.198 ve sonrasında subagent'lar varsayılan olarak arka planda (background) çalışır — ana konuşma subagent'ın bitmesini beklemeden devam edebilir. Claude, subagent'ın sonucuna hemen ihtiyaç duyduğunda o subagent'ı ön planda (foreground) çalıştırmayı kendisi seçer; bu senin elle ayarlaman gereken bir şey değildir.

🔑 ÖnemliBackground varsayılan davranıştır ama garanti değildir — Claude, sonucu bir sonraki adımda kullanması gerektiğini fark ederse subagent'ı foreground'a alır. Bunu bir agent'ın 'yavaş' çalıştığını düşünmeden önce hatırla.

KÜTÜPHANE🧰 Prompt & Şablon Kütüphanesi

1) code-reviewer Agent'ı Oluşturma
Bana .claude/agents/code-reviewer.md adında bir subagent oluştur. Görevi: kod değişikliklerini güvenlik ve kalite açısından incelemek. Sadece Read, Grep ve Glob tool'larına erişimi olsun — dosya değiştirme yetkisi olmasın. description alanını, değişiklik yapıldıktan sonra otomatik delegasyona uygun, spesifik bir cümleyle yaz.
İlk agent'ı bilinçli olarak salt-okunur tasarlıyoruz — least-privilege prensibinin en basit uygulaması.
2) Doğal Dil ile Delegasyon
code-reviewer agent'ını kullanarak son commit'teki değişiklikleri incele ve bulgularını önem sırasına göre listele.
Bu bir öneri niteliğindedir — Claude uygun görürse delege eder, garanti değildir.
3) @-mention ile Kesin Çağırma
@code-reviewer son commit'teki değişiklikleri incele ve bulgularını önem sırasına göre listele.
@-mention, belirtilen subagent'ın bu görev için kesinlikle çalışmasını garanti eder.
4) test-writer Agent'ı Oluşturma
Bana .claude/agents/test-writer.md adında bir subagent oluştur. Görevi: eksik test kapsamını bulup edge case'leri kapsayan testler yazmak. tools alanına Read, Grep, Glob, Edit ve sadece test çalıştırma komutlarına izin veren kısıtlı bir Bash kuralı ekle — genel Bash erişimi verme.
Bu agent dosya değiştirebilmeli ama komut çalıştırma yetkisi kısıtlı kalmalı.
5) Zincirleme Delegasyon
code-reviewer ile bulunan sorunlardan en az birini seç, sonra test-writer ile bu sorunu ortaya çıkaran/kapsayan bir test yazdır. Testin gerçekten çalıştığını doğrula.
İki agent'ı ardışık çağırarak reviewer bulgularını tester'ın girdisi haline getiriyorsun.

CHALLENGE🏆 Günün Challenge'ı

Todo App İçin 3 Uzman Agent Kur

Todo App (Gün 10) için üç agent tanımla — reviewer, tester, documenter — her birini least-privilege prensibiyle tasarla ve projeyi her biriyle analiz ettir.

Gereksinimler:

  • 3 agent dosyası .claude/agents/ altında tanımlanmış olmalı
  • reviewer salt-okunur (Read/Grep/Glob) kalmalı
  • tester yalnızca test yazma/çalıştırma için gerekli minimum tool setine sahip olmalı (genel Bash yok)
  • documenter dosya okuma ve dokümantasyon yazma izinleriyle sınırlı olmalı
  • Her agent'ın description alanı otomatik delegasyona uygun ve spesifik yazılmış olmalı
  • En az bir agent gerçek bir bulgu/çıktı üretmiş olmalı

Başarı Kriterleri:

✅ Çözüm Rehberi

Bu challenge'ı Prompt Kütüphanesi'ndeki desenleri izleyerek üç kez tekrarla: önce agent'ı oluştur, sonra description'ının otomatik delegasyonu tetikleyip tetiklemediğini test et, gerekirse description'ı netleştir.

Önerilen prompt zinciri:

1. reviewer
code-reviewer promptu ile salt-okunur bir inceleme agent'ı oluştur, çağır ve bulguları oku.
2. tester
test-writer promptu ile kısıtlı Bash kurallı bir test agent'ı oluştur, reviewer bulgularından birini test ettir.
3. documenter
Sadece dosya okuma ve markdown yazma iznine sahip bir documenter agent'ı oluştur; projenin güncel README'sini bu agent'a güncelletir.

Kontrol noktaları:

  • Her agent'ı oluşturduktan sonra description'ı test et: doğal dilde belirsiz bir istek verip Claude'un doğru agent'ı seçip seçmediğine bak.
  • tools alanını boş bırakmak 'her şeyi yapabilir' değil, parent'ın tool setini devralmak anlamına gelir — yine de bilinçli olarak kısıtlama yaz.
  • documenter'a Bash verme; sadece Read + Edit/Write yeterlidir, komut çalıştırma riski gereksiz.

Sık yapılan hatalar:

  • ⚠️ ⚠️ description'ı çok genel yazmak ("kod ile ilgilenir") — otomatik delegasyon güvenilir çalışmaz, Claude hangi agent'ı seçeceğini kestiremez.
  • ⚠️ ⚠️ tester'a sınırsız Bash vermek — kısıtlı Bash(pytest *) gibi kurallar yerine genel erişim, least-privilege'i bozar.
  • ⚠️ ⚠️ --agent ile tek görevlik delegasyonu karıştırmak — --agent tüm oturumu değiştirir, tek seferlik çağrı için doğal dil veya @-mention kullanılır.
  • ⚠️ ⚠️ Yeni oluşturulan .claude/agents/ klasörünün aynı oturumda hemen algılanacağını varsaymak — klasör oturumdan önce yoktuysa restart gerekir.

ÖZET📌 Takeaway'ler

  1. Subagent'ın asıl değeri context izolasyonudur: ara çıktıların (log, arama, dosya) ana konuşmayı doldurmasını engeller
  2. Frontmatter'da sadece name ve description zorunludur; tools verilmezse parent'ın tool setini devralır
  3. tools/disallowedTools "neye erişebilir", permissionMode "nasıl onay ister" sorularını ayrı ayrı cevaplar
  4. Doğal dil öneri, @-mention garanti, --agent ise tüm oturumu o agent kimliğiyle başlatan session-wide bir moddur
  5. Model çözümleme sırası: CLAUDE_CODE_SUBAGENT_MODEL → invocation model → frontmatter model → ana oturum modeli
  6. Nested subagent için Agent tool'unun tools listesinde olması gerekir; derinlik limiti 5'tir
  7. v2.1.198+ subagent'lar varsayılan background çalışır, Claude sonuca ihtiyaç duyunca foreground'a alır
  8. Subagent tek oturum içi delegasyon içindir; birden fazla bağımsız oturumu koordine etmek Agent Teams'in (Gün 17) işidir
