Hooks ve Sandboxing

“Claude her adım attığında sen de bir şey yap”

Dün subagent'larla Claude'un işi kime devredeceğine karar vermesini öğrendin. Bugün farklı bir soruna eğiliyorsun: Claude bir şey yaparken (dosya değiştirirken, komut çalıştırırken, bitirdiğini düşünürken) senin de her seferinde, modelin hatırlamasına bağlı olmadan devreye girecek deterministik bir mekanizma. Hooks, Claude Code'un yaşam döngüsündeki belirli noktalarda otomatik olarak tetiklenen script/istek/agent çağrılarıdır. Sandboxing ise bunun tamamlayıcısı: Claude otonom çalışırken Bash komutlarının işletim sistemi seviyesinde neye dokunabileceğini sınırlayan bir güvenlik katmanı. İkisi birlikte, 'ne olursa olsun her seferinde çalışsın' ve 'yanlış giderse bile zararı sınırlı kalsın' sorularına cevap verir.

BÖLÜM 1TEORİK TEMEL

1.1 Hook Nedir?

Bir hook, Claude Code'un yaşam döngüsündeki belirli noktalarda otomatik tetiklenen kullanıcı tanımlı bir event handler'dır — shell komutu, HTTP isteği, MCP tool çağrısı, model prompt'u veya bir subagent olabilir. Hook'lar modelin 'formatı çalıştırmayı hatırlamasına' bağlı değildir: event gerçekleştiği her seferinde, koşulsuz çalışırlar. Bu, hook'ları CLAUDE.md'deki bir kuraldan temelden farklı kılar — CLAUDE.md tavsiye niteliğindedir, hook zorlayıcıdır.

🔑 ÖnemliCLAUDE.md kuralları modelin dikkatine bağlıdır ('advisory'); hook'lar ise deterministik olarak her seferinde tetiklenir ('enforced'). Kritik bir kuralı (format, güvenlik, test) gerçekten garanti altına almak istiyorsan onu hook olarak yaz, sadece CLAUDE.md'ye yazma.

1.2 Event Taksonomisi

Claude Code'un resmi dokümantasyonu geniş bir event seti tanımlar (30'dan fazla) — SessionStart/SessionEnd, PreToolUse/PostToolUse, UserPromptSubmit, Stop/SubagentStop, PermissionRequest, TaskCreated/TaskCompleted, PreCompact, WorktreeCreate ve daha fazlası. Bugün hepsini ezberlemeyeceksin — pratikte en çok kullanacakların küçük bir alt kümesi var.

EventNe zaman tetiklenir
PreToolUseClaude bir tool çağırmadan hemen önce — bloklayabilir/değiştirebilir
PostToolUseTool başarıyla çalıştıktan sonra — geri alamaz, sadece tepki verir
UserPromptSubmitKullanıcı prompt gönderdiğinde, Claude işlemeden önce
StopClaude, yanıtını bitirdiğini düşündüğünde
SubagentStart / SubagentStopBir subagent başladığında / bitirdiğinde
SessionStart / SessionEndOturum başladığında / sonlandığında
TaskCreated / TaskCompletedGörev takibi oluşturulduğunda / tamamlandığında

1.3 Beş Handler Tipi

Bir hook 'tetiklendiğinde' aslında beş farklı tipte handler çalıştırılabilir. İlk ikisi (command, http) deterministik script/servis çağrılarıdır; son ikisi (prompt, agent) muhakeme gerektiren, regex/script ile ifade edilmesi zor kararlar için bir Claude modeli kullanır. mcp_tool ise bağlı bir MCP server'ın tool'unu doğrudan hook olarak çağırmayı sağlar.

TipNe yaparGirdi/Çıktı
commandShell komutu/script çalıştırırstdin'den JSON alır, stdout/exit code ile cevap verir
httpBir HTTP endpoint'e POST ederEvent JSON'u POST body'de gider, aynı JSON formatında cevap döner
mcp_toolBağlı bir MCP server'ın tool'unu çağırırÖrn. PostToolUse'da bir güvenlik tarama tool'u çalıştırma
promptTek turlu bir Claude modeline (varsayılan Haiku) değerlendirtirPrompt metninde $ARGUMENTS ile hook input JSON'u modele gösterilir
agentTool erişimi olan bir subagent doğurur (çoklu tur)Dosya okuma/komut çalıştırma gerektiren derin doğrulamalar için
🔑 Önemli$ARGUMENTS placeholder'ı sadece prompt ve agent handler'ların prompt metninde geçerlidir — hook input JSON'unu modele göstermek için kullanılır. command hook'larda girdi stdin'den, http hook'larda POST body'den gelir; $ARGUMENTS orada bir kavram değildir.
💡 İpucuBugün ve çoğu pratik senaryoda command handler'ı yeterlidir — hızlı, ücretsiz ve öngörülebilir. prompt/agent'a, kararın 'bu commit mesajı conventional format'a uyuyor mu' gibi regex ile ifade edilemeyecek kadar yargı gerektirdiği durumlarda geç.

1.4 Matcher vs if

İki farklı filtreleme katmanı var ve karıştırılmamalı. matcher, hook'un hangi event/tool için tetikleneceğini belirler (ör. sadece Bash tool'u). if ise permission-rule sözdizimiyle (Bash(rm *), Edit(*.ts) gibi) tool'un argümanlarına bakarak daha ince bir filtre uygular — hook process'inin gereksiz yere spawn edilmesini önler.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {"type": "command", "if": "Bash(rm *)",
           "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/block-rm.sh"}
        ]
      }
    ]
  }
}
⚠️ Dikkatif filtresi best-effort'tur, hard security gate değildir. Bash komutu parse edilemezse filtre fail-open olur — yani belirsizlik durumunda hook yine de çalışır (bu güvenlik tarafında iyi bir varsayılan). Ama tersi yönde net ol: 'if' ile 'bu asla tetiklenmesin' garantisi veremezsin. Sert allow/deny garantisi için hook değil, permission sistemindeki deny rule'ları kullan.

1.5 Exit Code ve JSON Decision

Bir command hook'unun Claude Code'a geri bildirim vermesinin iki yolu vardır: basit exit code veya yapılandırılmış JSON çıktısı. Basit olan çoğu durumda yeterlidir; JSON çıktısı daha ince kontrol (belirli bir sebep, kısmi izin, girdi değiştirme) gerektiğinde devreye girer.

Exit codeAnlamı
0Başarı; stdout varsa Claude Code onu JSON olarak parse etmeyi dener
2Bloklayıcı hata; PreToolUse için tool çalışmaz, stderr Claude'a 'neden bloklandığı' olarak gösterilir
diğerBloklayıcı olmayan hata; aksiyon devam eder, hata verbose modda/transcript'te görünür

PreToolUse için daha ince kontrol istiyorsan JSON çıktısında hookSpecificOutput.permissionDecision alanını dört değerden biriyle döndürebilirsin: allow, deny, ask, defer.

🔑 ÖnemliBunu bir 'ekstra izin mekanizması' olarak düşünme — pratikte hook'lar defense-in-depth'in doğrulama/bloklama katmanıdır. Hook sessiz kalırsa (çıktı yoksa) normal permission akışı olduğu gibi devam eder; bu otomatik onay anlamına gelmez. Ve birden fazla hook farklı kararlar dönerse, deny her zaman allow'u geçersiz kılar — bir hook asla permission sisteminin zaten vereceğinden daha fazla erişim açamaz.

1.6 Stop Hook ve Sonsuz Döngü Koruması

Bir Stop hook'u, Claude yanıtını bitirdiğini düşündüğünde 'hayır, devam et' diyebilir (exit 2 veya decision: block). Ama bunu dikkatsizce yazarsan sonsuz bir döngüye girersin: Claude her bitirdiğinde hook tekrar 'devam et' der. Bunu önlemek için hook, girdi JSON'undaki stop_hook_active alanını kontrol edip true ise erken çıkmalıdır.

#!/bin/bash
INPUT=$(cat)
if [ "$(echo "$INPUT" | jq -r '.stop_hook_active')" = "true" ]; then
  exit 0   # Claude'un durmasına izin ver
fi
# ... asıl kontrol mantığı (ör. npm test) ...
💡 İpucuKendi guard'ını unutsan bile Claude Code bir üst sınır uyguluyor: bir Stop hook'u ilerleme kaydetmeden art arda 8 kez bloklarsa Claude Code o hook'u geçersiz kılıp turu bitirir. Bu eşik CLAUDE_CODE_STOP_HOOK_BLOCK_CAP ile yükseltilebilir — ama guard'ı doğru yazmak yine de senin sorumluluğun.

1.8 Üç Katmanlı Güvenlik Modeli

Bugünün üç kavramını (permissions, hooks, sandbox) birbirinin alternatifi değil, birbirini tamamlayan katmanlar olarak gör:

KatmanSoruyu cevaplar
PermissionsBu tool hiç çalışabilir mi?

🔑 ÖnemliPermissions hangi tool'un çalışabileceğine karar verir; hooks lifecycle event'lerinde deterministik doğrulama/otomasyon yapar; sandbox ise özellikle Bash ve alt süreçlerinin işletim sistemi seviyesinde neye erişebileceğini sınırlar. Bölüm 3'te sandbox'ı işlerken bu tabloya geri döneceğiz.

BÖLÜM 2PRATİK — ADIM ADIM

2.1 İlk Hook: Otomatik Format (PostToolUse)

En basit ve en hemen faydalı hook, bir dosya yazıldığında/düzenlendiğinde otomatik formatlama çalıştırmaktır. Bu bir PostToolUse hook'udur — tool zaten çalıştı, sen sadece tepki veriyorsun, geri alamazsın.

  1. Todo App proje dizininde Claude Code'u başlat: claude
  2. Prompt Kütüphanesi'ndeki 1. promptu ver: PostToolUse formatter hook'u yazdır
  3. Oluşan .claude/settings.json içindeki hooks.PostToolUse girdisini incele
  4. Bir dosyayı Claude'a değiştirt, formatter'ın otomatik çalıştığını gözlemle
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {"type": "command",
           "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\""}
        ]
      }
    ]
  }
}

2.2 Güvenlik Hook'u: Tehlikeli Komut Engelleyici (PreToolUse)

Şimdi tersi yönde bir hook: tool çalışmadan önce devreye giren, bloklama yetkisi olan bir PreToolUse hook'u. rm -rf gibi yıkıcı komutları exit 2 ile durdurup nedenini Claude'a stderr üzerinden bildireceksin.

  1. Prompt Kütüphanesi'ndeki 2. promptu ver: dangerous command blocker hook'u yazdır
  2. Hook script'ini oku — komut içinde rm -rf / drop table gibi desenleri arıyor mu?
  3. Bilerek tehlikeli bir komut istet (ör. "rm -rf /tmp/test klasörünü sil") ve bloklandığını doğrula
  4. Blok mesajının Claude'a neden engellendiğini açıkça anlattığını kontrol et
⚠️ DikkatBir deny hook'u bypassPermissions modunda bile çalışır — kullanıcı normal onay akışını atlasa dahi PreToolUse deny kararı geçerlidir. Bu, kurumsal politikaların (ör. 'production dosyalarına asla yazma') atlanamaz olmasını sağlayan bilinçli bir tasarım.

2.3 /hooks ile Denetim ve Debug

Hook'ları yazdıktan sonra gerçekten kayıtlı olduklarını ve beklendiği gibi çalıştıklarını doğrulamak gerekir. /hooks komutu kayıtlı hook'ları event bazında listeler (salt-okunur bir görünümdür — değişiklik dosyadan yapılır); Ctrl+O verbose modu açıp hook çıktısını transcript'te gösterir.

  1. /hooks komutunu çalıştır, PostToolUse ve PreToolUse altında hook'larının göründüğünü doğrula
  2. Ctrl+O ile verbose mode'a geç
  3. Bir dosya düzenlemesi yaptır, hook'un stdout/stderr çıktısının transcript'te göründüğünü gözlemle
  4. Gerekirse claude --debug ile hangi hook'ların eşleştiğini ve exit code'larını incele
💡 İpucuBir hook manuel olarak test edilebilir: örnek JSON'u stdin'den ver ve exit code'u kontrol et — echo '{"tool_name":"Bash","tool_input":{"command":"ls"}}' | ./my-hook.sh && echo $?

BÖLÜM 3PRATİK — DERİNLEŞ

3.1 Stop Hook: Test-Gate

Şimdi bugünün en güçlü desenlerinden birini kuracaksın: Claude 'bitirdim' dediğinde otomatik olarak test suite'ini çalıştıran bir Stop hook'u. Testler geçmezse Claude'u durdurmadan devam ettirebilirsin.

  1. Prompt Kütüphanesi'ndeki 3. promptu ver: stop_hook_active guard'lı bir test-gate hook'u yazdır
  2. Script'in stop_hook_active kontrolünü doğru yaptığını incele
  3. Claude'a bilerek testleri bozacak bir değişiklik yaptır, Stop hook'unun devam ettirdiğini gözlemle
  4. Testleri düzelttir, hook'un artık durmasına izin verdiğini doğrula
🔑 ÖnemliBu desen, 'test yaz ve çalıştır' talimatını CLAUDE.md'de bırakmak yerine gerçekten zorunlu kılar — Claude bitirdiğini düşünse bile testler geçmeden tur bitmez.

3.2 Sandboxing'e Giriş: /sandbox

Sandboxing, hook'lardan farklı bir katmandır: hook'lar 'ne zaman ne kontrol edilsin' sorusuna cevap verirken, sandbox 'Bash çalıştıktan sonra neye dokunabilsin' sorusuna cevap verir. /sandbox komutuyla etkinleştirilir ve bir permission mode değildir — iki çalışma biçimi sunar: auto-allow (sandbox sınırları içindeki komutlar onaysız çalışır) ve regular permissions (sandbox sınırları aynı ama normal onay akışı korunur).

⚠️ DikkatOrtam notu: macOS'ta ek kurulum gerekmez (Seatbelt kullanılır). Linux/WSL2'de bubblewrap ve socat paketleri gerekir. Native Windows desteklenmiyor — Windows'ta Claude Code'u WSL2 içinde çalıştırman gerekir.

3.3 Filesystem İzolasyonu — Write ve Read Farklı Davranır

Sandbox açıkken varsayılan davranış write ve read için simetrik değildir — bu, dersin en kritik güvenlik notudur:

ErişimVarsayılan kapsam
WriteSadece çalışma dizini (CWD) + oturum temp dizini; dışına yazmak engellenir
ReadTüm bilgisayar — ~/.aws/credentials, ~/.ssh/ gibi credential dosyaları dahil
⚠️ DikkatSandbox tek başına credential okumasını engellemez! Hassas dosyaları korumak için sandbox.filesystem.denyRead ile açıkça engellemen ya da sandbox.credentials ile ortam değişkenlerini maskeleyen bir konfigürasyon eklemen gerekir. Bu izolasyonu 'her şeyi kapatır' diye varsaymak, dersin en yaygın yanlış anlaşılmasıdır.

3.4 Network İzolasyonu

Ağ erişimi, sandbox dışında çalışan bir proxy server üzerinden kontrol edilir: izin verilen domain listesi tanımlanır, yeni bir domain'e istek gittiğinde kullanıcıdan onay istenir. Alt süreçler (npm, pip gibi paket yöneticileri dahil) de aynı kısıtlamayı miras alır — bir build script'inin spawn ettiği alt süreç kuralları atlayamaz.

💡 İpucuFilesystem ve network izolasyonu birlikte anlamlıdır: sadece filesystem korunursa saldırgan ağ üzerinden veri sızdırabilir; sadece network korunursa saldırgan sistem ikili dosyalarını değiştirerek ağ erişimi kazanabilir. İkisi birlikte kullanılmalı.

3.5 Sandbox vs Permissions — Tamamlayıcı Katmanlar

Permissions tüm tool'lara (Bash, Read, Edit, WebFetch, MCP tool'ları dahil) uygulanır ve 'bu tool hiç çalışsın mı' sorusuna cevap verir. Sandbox ise sadece Bash ve onun alt süreçlerini kapsar ve 'çalıştıktan sonra neye dokunabilsin' sorusuna cevap verir. Built-in Read/Edit/Write tool'ları sandbox'ın değil, doğrudan permission sisteminin kapsamındadır.

  1. /sandbox çalıştır, auto-allow modunu seç
  2. Claude'dan çalışma dizini dışına bir dosya yazmasını iste (ör. üst dizine touch)
  3. İşletim sistemi seviyesinde reddedildiğini (Operation not permitted) doğrula
  4. Aynı denemeyi sandbox kapalıyken tekrarla, farkı gözlemle

KÜTÜPHANE🧰 Prompt & Şablon Kütüphanesi

1) PostToolUse Format Hook'u
Todo App projesinde her Write veya Edit işleminden sonra otomatik olarak prettier (veya projenin kullandığı formatter) çalıştıran bir PostToolUse hook'u .claude/settings.json içine ekle. Hook'u ekledikten sonra bir dosyayı düzenleyip gerçekten çalıştığını doğrula.
Bugünün en basit ve en hemen faydalı hook'u — ana konuşmayı 'prettier çalıştır' hatırlatmasından kurtarır.
2) Tehlikeli Komut Engelleyici
rm -rf, DROP TABLE ve benzeri yıkıcı komut desenlerini tespit edip exit code 2 ile bloklayan bir PreToolUse hook'u yaz. Bash tool'una matcher ile bağla, stderr mesajının Claude'a neden engellendiğini net anlatmasını sağla.
Bu hook bypassPermissions modunda bile çalışır — kurumsal güvenlik politikasının teknik temelidir.
3) Stop Hook ile Test-Gate
Claude her yanıtını bitirdiğinde otomatik test suite'imi çalıştıran bir Stop hook'u yaz. Testler geçmezse Claude'u devam ettir (exit 2 + açıklama), geçerse durmasına izin ver. Sonsuz döngüyü önlemek için stop_hook_active alanını kontrol eden bir guard ekle.
stop_hook_active guard'ı olmadan bu hook sonsuz döngüye girer — mutlaka ekle.
4) Sandbox'ı Etkinleştirip Doğrulama
/sandbox komutunu çalıştır, auto-allow modunu seç. Sonra benden çalışma dizinim dışına bir test dosyası oluşturmamı iste ve bunun engellendiğini bana göster. Ardından ~/.ssh/ dizinini okumayı dene ve varsayılan read davranışının bunu engelleyip engellemediğini raporla.
Bu prompt bilerek write VE read davranışını ayrı ayrı test ettiriyor — ikisinin farklı olduğunu deneyimlemek için.

CHALLENGE🏆 Günün Challenge'ı

Todo App'e Hook Seti + Sandbox Doğrulaması

Todo App'e (Gün 10) üç hook ekle — güvenlik, format, test-gate — ve sandbox ile izolasyonun gerçekten çalıştığını kanıtla.

Gereksinimler:

  • .claude/settings.json içinde en az 3 hook tanımlı olmalı: PreToolUse (dangerous command blocker), PostToolUse (format), Stop (test-gate)
  • Dangerous command blocker rm -rf veya benzeri bir komutu gerçekten bloklamalı
  • Stop hook'u stop_hook_active guard'ı içermeli
  • /hooks ile üç hook'un da doğru event altında göründüğü doğrulanmalı
  • /sandbox etkinleştirilip CWD dışına yazma denemesi engellenmiş olmalı
  • Sandbox'ın sadece Bash'i kapsadığı, Read/Edit/Write'ın permission sistemiyle yönetildiği açıklanmalı

Başarı Kriterleri:

✅ Çözüm Rehberi

Bu challenge'ı üç aşamada ilerlet: önce hook'ları tek tek ekle ve her birini izole test et, sonra hepsini birlikte bir gerçek senaryoda (dosya düzenleme + test + commit) dene, son olarak sandbox'ı açıp aynı projede izolasyonu doğrula.

Önerilen prompt zinciri:

1. Güvenlik hook'u
PreToolUse dangerous command blocker'ı ekle, rm -rf ile test et, blok mesajını oku.
2. Format hook'u
PostToolUse formatter'ı ekle, bir dosyayı düzenlet, otomatik formatlandığını doğrula.
3. Test-gate hook'u
Stop hook'unu stop_hook_active guard'ıyla ekle, testleri bilerek bozup düzelterek davranışı gözlemle.
4. Sandbox doğrulaması
/sandbox aç, CWD dışına yazmayı ve ~/.ssh/ okumayı dene, write/read farkını raporlat.

Kontrol noktaları:

  • Her hook'u eklendikten hemen sonra izole test et — üçünü birden yazıp sonda test etmek hangi hook'un neden çalışmadığını bulmayı zorlaştırır.
  • Stop hook'unu test ederken önce guard'sız bir versiyonla (kontrollü şekilde) sonsuz döngüyü gözlemlemek, guard'ın neden gerekli olduğunu somutlaştırır.
  • Sandbox denemesinde write ve read'i ayrı ayrı test et — ikisinin farklı sonuç vermesi bu dersin en önemli 'aha' anıdır.

Sık yapılan hatalar:

  • ⚠️ ⚠️ Stop hook'una stop_hook_active guard'ı eklemeyi unutup sonsuz döngüye girmek.
  • ⚠️ ⚠️ if filtresini 'kesin güvenlik' sanıp permission deny rule'larını atlamak — if best-effort'tur, fail-open olabilir.
  • ⚠️ ⚠️ Sandbox'ın credential dosyalarını da okunmaz yapacağını varsaymak — varsayılan read sınırsızdır, denyRead açıkça eklenmelidir.
  • ⚠️ ⚠️ PreToolUse hook'unun sadece Bash'i değil Read/Edit/Write'ı da sandbox kapsadığını sanmak — bu tool'lar permission sisteminin, sandbox'ın değil.
  • ⚠️ ⚠️ Hook'u settings.json'a elle ekleyip aynı oturumda /hooks ile görünmesini beklemek — bazı değişiklikler için reload/restart gerekebilir.

ÖZET📌 Takeaway'ler

  1. Hook'lar deterministiktir — modelin hatırlamasına değil, event'in gerçekleşmesine bağlıdır; CLAUDE.md tavsiye, hook zorlayıcıdır
  2. 5 handler tipi var: command, http, mcp_tool, prompt, agent — pratikte command ağırlıklı, diğerleri yargı gerektiren durumlar için
  3. matcher event/tool seviyesinde filtreler, if ise argüman seviyesinde filtreler ama best-effort'tur — hard security için permission deny rule kullanılır
  4. PreToolUse deny, bypassPermissions modunda bile geçerlidir; hook sessiz kalırsa normal permission akışı devam eder, bu otomatik onay değildir
  5. Stop hook'ları stop_hook_active guard'ı gerektirir; guard'sız kalırsa Claude Code 8 ardışık bloktan sonra hook'u geçersiz kılar
  6. Sandbox varsayılan olarak write'ı CWD+temp ile sınırlar ama read'i tüm bilgisayara açık bırakır — credential koruması için denyRead şart
  7. Sandbox sadece Bash ve alt süreçlerini kapsar; Read/Edit/Write built-in tool'ları permission sisteminin kapsamındadır
  8. Permissions/hooks/sandbox birbirinin alternatifi değil, defense-in-depth katmanlarıdır: hangi tool çalışabilir → ne kontrol edilmeli → çalıştıktan sonra neye dokunabilir
